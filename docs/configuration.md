# Configuration reference

The gateway has two configuration layers:

- `.env` contains machine-local values and secrets. It is ignored by Git.
- `config.yaml` contains shareable LiteLLM routes and behavior.

Keeping these separate makes a configuration safe to review and commit without making secrets repetitive or platform-specific.

Command examples assume the project virtual environment is active:

```bash
# macOS/Linux
source .venv/bin/activate

# Windows Git Bash
source .venv/Scripts/activate
```

```powershell
# Windows PowerShell
.venv\Scripts\Activate.ps1
```

## Precedence

From highest to lowest priority:

1. Launcher flags such as `--host`, `--port`, and `--config`.
2. Variables already present in the parent process environment.
3. Variables loaded from `.env`.
4. Built-in launcher defaults.

`python-dotenv` is intentionally called with `override=False`, so a CI system or secret manager can inject a value without a local file replacing it.

## Environment variables

### Provider and proxy credentials

| Variable | Required | Description |
| --- | ---: | --- |
| `GEMINI_API_KEY` | For the default route | Gemini API key created in Google AI Studio |
| `LITELLM_MASTER_KEY` | Yes | Local bearer token accepted by LiteLLM; generated by setup |

Provider examples add their own variables, such as `OPENAI_API_KEY`. Use names expected by the
LiteLLM provider adapter and reference them as `os.environ/VARIABLE_NAME` in YAML.
The doctor and launcher discover these references recursively and validate their presence, so
adding a provider does not require a Python change.

### Launcher behavior

| Variable | Default | Description |
| --- | --- | --- |
| `GATEWAY_MODEL` | `gemini-3.1-flash-lite` | LiteLLM alias for Claude Code's main model |
| `GATEWAY_SMALL_MODEL` | main model | Alias for fast/background requests |
| `GATEWAY_HOST` | `127.0.0.1` | LiteLLM listen address |
| `GATEWAY_PORT` | `4000` | LiteLLM listen port, from 1 to 65535 |
| `GATEWAY_CONFIG` | `config.yaml` | Absolute path or clone-relative YAML path |

The launcher requires both model variables to match a `model_name` entry. This catches typos
before Claude Code starts.

### Variables passed to Claude Code

The `claude` command builds a child-only environment. It does not persist global settings.

| Claude Code variable | Value |
| --- | --- |
| `ANTHROPIC_BASE_URL` | Local gateway URL |
| `ANTHROPIC_AUTH_TOKEN` | `LITELLM_MASTER_KEY` |
| `ANTHROPIC_MODEL` | `GATEWAY_MODEL` |
| `ANTHROPIC_SMALL_FAST_MODEL` | `GATEWAY_SMALL_MODEL` |
| `ANTHROPIC_DEFAULT_OPUS_MODEL` | `GATEWAY_MODEL` |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | `GATEWAY_MODEL` |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | `GATEWAY_SMALL_MODEL` |

Setting both legacy/general and model-family overrides supports Claude Code versions that select
models in different ways. Every name still resolves through LiteLLM's model list.

The LiteLLM child intentionally does not inherit generic `DEBUG` or `DETAILED_DEBUG` variables.
Other developer tools commonly use `DEBUG` for non-boolean values, while LiteLLM interprets it as
a boolean CLI option. Suppressing both prevents accidental startup failures and sensitive detailed
request logging.

`PYTHONUTF8=1` and `PYTHONIOENCODING=utf-8` are set for the LiteLLM child so its Unicode startup
output also works in Windows pipes and CI logs that would otherwise use a legacy code page.

## `config.yaml`

```yaml
model_list:
  - model_name: gemini-3.1-flash-lite
    litellm_params:
      model: gemini/gemini-3.1-flash-lite
      api_key: os.environ/GEMINI_API_KEY

litellm_settings:
  drop_params: true

general_settings:
  master_key: os.environ/LITELLM_MASTER_KEY
```

### `model_list`

Each item is one route:

- `model_name` is the stable public alias clients request. It need not equal a provider model ID.
- `litellm_params.model` is LiteLLM's `provider/model` identifier.
- `litellm_params.api_key` resolves a credential from the process environment. Never put the
  literal key here.

Provider adapters accept additional fields such as `api_base`, `api_version`, `vertex_project`,
or `vertex_location`. Only add fields required by that provider. Consult LiteLLM's current
provider documentation because adapters evolve.

Multiple entries may share a `model_name`; LiteLLM can then route across deployments. That is
useful for load balancing but adds operational complexity, so the default remains one alias and
one deployment.

### `litellm_settings.drop_params`

When `true`, LiteLLM drops parameters that the target provider does not support. Claude Code can
send Anthropic-specific fields, making this useful for cross-provider translation. The trade-off
is that an unsupported feature may be omitted instead of rejected. Test tool use and model
behavior when adding a provider.

### `general_settings.master_key`

Protects LiteLLM endpoints with a bearer token. The value resolves from `.env`. A hard-coded
shared token would make every clone predictable, so setup generates a random token per clone.

## Command options

```text
claude-gateway [--config PATH] setup [--force]
               [--configure-vscode | --no-configure-vscode]
claude-gateway [--config PATH] configure-vscode
               [--scope {local,user}] [--dry-run] [--remove]
               [--host HOST] [--port PORT]
claude-gateway [--config PATH] doctor [--host HOST] [--port PORT]
claude-gateway [--config PATH] start [--host HOST] [--port PORT]
claude-gateway [--config PATH] claude [--host HOST] [--port PORT]
               [--timeout SECONDS] [-- CLAUDE_ARGS...]
```

- `setup --force` asks for replacement values for every credential referenced by the selected
  YAML file and rotates the local proxy token.
- Interactive `setup` offers to configure project-local Claude Code settings. Use
  `--configure-vscode` or `--no-configure-vscode` to answer non-interactively.
- `configure-vscode` merges gateway settings into `.claude/settings.local.json` by default.
  `--scope user` targets `~/.claude/settings.json` instead.
- `configure-vscode --dry-run` reports the target, model, URL, and a redacted token without
  changing files. `--remove` removes only gateway-owned model and environment entries.
- `claude --timeout` controls how long the launcher waits for LiteLLM readiness (default 60s).
- Arguments after `--` are forwarded unchanged to Claude Code.

Global options must appear before the subcommand; for example:

```bash
claude-gateway --config config.local.yaml doctor
```

## Generated Claude Code settings

The VS Code wizard manages these values:

- `ANTHROPIC_BASE_URL`
- `ANTHROPIC_AUTH_TOKEN`
- `ANTHROPIC_DEFAULT_OPUS_MODEL`
- `ANTHROPIC_DEFAULT_SONNET_MODEL`
- `ANTHROPIC_DEFAULT_HAIKU_MODEL`
- `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY`
- top-level `model`

All other settings and `env` entries are preserved. Before changing an existing file, the wizard
copies it to the same filename with `.bak` appended. Both project-local files are ignored because
they contain the proxy token. Re-running the command with the same configuration is a no-op.

## Local overrides

For significant local changes, copy `config.yaml` to `config.local.yaml` and add
`GATEWAY_CONFIG=config.local.yaml` to `.env`. The local file is ignored, so upstream updates to
the default remain easy to review. For changes useful to everyone, update `config.yaml` and open
a pull request.
